# 三方辩论 — 设计笔记

## 为什么不用 LeSingh1/debate-skill 的方案

| 维度 | LeSingh1 | 三方辩论 |
|------|----------|---------|
| 智能体数 | 5（Visionary/Skeptic/Engineer/Economist/Ethicist） | 3（激进/保守/中立） |
| 调用模式 | 串行 16 次 API 调用（4 分钟） | 并行 6 次（2 轮 × 3） |
| 编排器 | Python 脚本直接调 Claude API | Hermes `delegate_task` 原生 |
| 平台 | Claude Code 专用 | Hermes Agent 原生 |
| 输出语言 | 英文 | 中文 |
| 立场设置 | 职业/功能角色 | 意识形态/风险偏好 |
| 适合场景 | 产品/技术决策 | 战略/人生/两难决策 |

## 为什么选 激进/保守/中立 而非专家人格

- 专家人格（Visionary/Skeptic/...）适合**技术决策**——每个人格代表一个功能视角
- 激进/保守/中立适合**两难决策**——核心是风险偏好和价值观冲突
- 中文用户习惯 "左中右" 或 "激进保守" 的思维框架
- 三方数量刚好：够多元又不会冗余。5 方容易导致观点重复

## 为什么用 delegate_task 而非 API 调用

- Hermes 原生，无需额外配置 API key
- 自动继承当前会话的模型和 provider
- 子智能体的系统 prompt 通过 `context` 注入，简单可控
- 并行执行天然支持，不需要自己管理并发

## 关键设计约束

1. **子智能体 context 要完整**：子智能体不知道其他人在说什么，必须把所有需要的上下文（对方观点、人设）都塞进 `context` 字段
2. **语言约束要显式**：子智能体默认可能用英文，`context` 末尾必须加 "用中文回复"
3. **字数限制**：每轮 200-400 字，防止泛泛而谈
4. **不和稀泥**：最终裁决必须有明确倾向 + "什么情况下反过来"的边界条件

## 与 ai-expert-consultation 的区别

`ai-expert-consultation` 也是多视角咨询，但：
- 它咨询的是**不同 AI 模型**（GPT-5.5 / Claude / Gemini 等），利用模型多样性
- 三方辩论是**同一模型的三种人格**，利用 prompt 多样性
- 专家咨询适合技术方案对比，三方辩论适合价值观/风险偏好型决策
- 两者互补，不重叠
